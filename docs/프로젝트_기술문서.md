# 프로젝트 기술 문서

최종 업데이트: 2026-05-24 +09:00 (KST)

이 문서는 현재 구현(Windows PowerShell 로컬 서버시간 측정 도구)의 구조·API·측정 알고리즘·성능/오차·운영 정책을 한곳에 정리한 기술 레퍼런스다. 사용자용 시각 자료는 `알고리즘_설명.html`(전체 흐름 + 방법1/2 비교)과 `사용설명서.html`에 있다. 과거 변경의 상세 로그는 루트 `CLAUDE.md`를 본다.

> 주의: `CLAUDE.md`에는 과거 NAS/Node.js 장기 설계안이 함께 남아 있다. 현재 동작 판단은 이 문서와 코드(`src/`)를 우선한다.

---

## 1. 개요

사용자가 입력한 웹사이트 URL의 서버 시각을 추정해 브라우저에 실시간 시계로 표시하는 PC용 로컬 도구다. 외부 배포형 서비스가 아니라 `ServerTimeProbe.exe` 또는 `run.bat`로 실행하는 로컬 HTTP 서버 + 정적 웹 UI 구조다. 목표 정확도는 좋은 조건에서 ±50ms 수준이다.

## 2. 실행 흐름과 주요 파일

1. `ServerTimeProbe.exe` 또는 `run.bat` 실행 → `src/probe.ps1`
2. 로컬 HTTP 서버를 `127.0.0.1:8765`에 띄우고 브라우저 자동 오픈
3. 상단 입력창에 URL 입력 → 서버가 적응형(약 6초) 샘플링으로 오프셋 계산
4. 웹 UI가 `/api/state` 기준 시각을 `performance.now()`로 보간해 표시

## 3. API

- `GET /` — 정적 UI. 첫 페이지 로드는 재측정을 걸지 않고, 같은 대상이 설정된 뒤 새로고침/재진입 시 재측정.
- `POST /api/target` — `{ "url": "https://example.com/" }`. 스킴 없으면 `https://` 부착, `http`/`https`만 허용.
- `GET /api/state` — 현재 측정 상태. 주요 필드:

| 필드 | 의미 |
|---|---|
| `host` / `targetUrl` | 현재 대상 |
| `offsetMs` | PC 기준 대비 서버 시각 오프셋 |
| `rttMedianMs` | 샘플 RTT 중앙값 |
| `sigmaMs` / `ci95Ms` | edge 중점 산포 / 보고 ± (교집합 계열은 일치 폭/2) |
| `method` | `edge-intersect` / `edge-intersect-robust` / `edge-median` / `upper-envelope` / `naver-time-api` |
| `acceptedCount` / `edgeCount` | 채택(합의) edge 수 / 전체 edge 수 |
| `intersectWidthMs` | 교집합 폭(feasible 영역) |
| `status` | `idle` / `queued` / `measuring` / `ok` / `failed` / `stale` |
| `lastRemeasureResult` | `accepted` / `kept-small-delta` / `rejected` / `failed-insufficient-edges` |
| `pcSendTimeAtMs` | 응답 직전 PC 기준 epoch ms |
| `ntpInfo` | 참고용 PC-NTP skew |

- `GET /api/samples` — 측정 상세 시각화용. 샘플 배열과 edge 배열(`lowerMs`/`upperMs`/`offsetMs`/`edgePcMs`), `method`, `intersectWidthMs` 등.

## 4. 측정 알고리즘

### 4.1 Cristian + 적응형 다중 샘플(순차)

한 샘플: `t1` 송신, `t2` 수신, `Ts`=Date 헤더 → `offset = Ts + RTT/2 − t2`. 샘플 루프는 **순차**(한 응답 받고 50ms 쉰 뒤 다음 요청)라 1회 비용이 `RTT + 50ms`다. 먼저 3회 RTT 프로브로 median RTT를 잰 뒤 `Count = ceil(6000/(RTT+50))`를 [10,60]로 클램프해 약 6초 윈도우를 유지한다.

병렬 발사를 안 하는 이유: (1) 서버 매너/차단 위험, (2) 동시 요청이 RTT를 혼잡 지연으로 오염시켜 `Ts+RTT/2` 가정이 깨짐, (3) edge detection이 시간순 샘플을 전제로 함.

per-sample timeout도 적응형 `max(1초, ceil(5×RTT))`(정수 초 하한, 캡 5초)으로, hang 1회가 윈도우를 통째로 먹는 것을 막는다.

### 4.2 HEAD vs GET

`HEAD` 우선, 실패 시 `GET + Range: bytes=0-0` 폴백(둘 다 `Date` 동일). HEAD는 일부 서버(구형 PHP/WAF)에서 hang될 수 있어 적응형 timeout으로 방어한다.

### 4.3 Edge detection → 오프셋 추정 (교집합 기본)

`Date` 헤더는 초 단위라 밀리초가 없다. 그래서 `Date`가 `N초→N+1초`로 바뀌는 **경계**를 인접 두 샘플 사이에서 잡는다. 각 edge는 오프셋 θ에 대한 구간 제약 `θ ∈ [S−R, S−L]`을 준다(L,R = 양 끝 서버이벤트 PC시각, S = 새 정수 초).

θ는 측정 내내 상수이고 서버 초 경계는 정확히 1초 간격이므로, **모든 구간의 교집합**이 곧 추정이다(균등노이즈의 MLE, edge 수 n에 대해 1/n 수렴). 모순(outlier)에 대비한 하이브리드:

- 전체 일치 → `edge-intersect`
- 일부 outlier 제외 후 일치(최대 겹침 ≥2) → `edge-intersect-robust`
- 어떤 두 edge도 안 겹침 → `edge-median`(중점들의 median 폴백)
- 경계 자체가 안 보임(캐시/이상) → `upper-envelope`(raw offset 상한봉투)

> 과거에는 "중점들의 median"이 기본이었으나(1/√n), 2026-05-24부터 1초 격자 교집합이 기본이다. 방법1(median) vs 방법2(교집합) 상세 비교는 `알고리즘_설명.html`을 본다.

RTT 필터(`Get-RttThreshold = rttMedian×1.5+2`)와 Gap 필터((rttMedian+50)×3)로 σ가 폭주하는 edge는 제외한다. edge가 부족하면(MinEdgeCount=8 미만) 윈도우를 최대 3회×3초 연장한다(`Test-ShouldExtendWindow`).

### 4.4 CDN 캐시 보정 (`Date + Age`)

CloudFront 같은 엣지 캐시 뒤의 사이트(예: `nol.interpark.com`)는 `Date`를 **캐시 적재 시각에 고정**하고, 그 뒤 경과 시간을 `Age` 헤더(초, RFC 9111)로 노출한다. 이때 `Date`만 보면 값이 안 변해 edge가 0개 → `upper-envelope` 폴백 + 정지된 시각이 나온다.

대응: 실효 서버시각을 **`Date + Age`** 로 계산한다(`Get-EffectiveServerDateMs`). `Age`는 서버 정수 초 경계에서 1씩 증가하므로 결합값은 1초 해상도의 라이브 클럭이 되고, 4.3의 edge 교집합 추정이 그대로 동작한다(ms 정밀 유지). `Age`가 없으면 0으로 취급 → `Date` 원본(일반 origin 무영향). 실측: `nol.interpark.com/ticket`에서 raw Date 초가 한 값에 고정인 동안 `Date+Age`는 `…13,14,15,16,17…`로 깔끔히 증가 → `upper-envelope`에서 `edge-intersect`로 복구.

> 주의: `Age`는 정수 초라 결합값은 초 해상도(일반 `Date`와 동일). RFC상 캐시는 origin의 `Date`를 보존하므로 라이브 `Date`+`Age`를 동시에 주는 비정상 프록시가 아닌 한 안전하다.

**범용성 검증(2026-05-24).** 7개 사이트 헤더 조사 결과 두 부류로만 갈렸고, 둘 다 올바르게 처리된다:

| 부류 | 사이트 | 거동 | 이 보정의 효과 |
|---|---|---|---|
| 일반 origin | google, github, cloudflare, interpark.com, ticket.interpark.com, melon티켓 | live `Date`, `Age` 없음 | `Age=0` → 무변화(기존과 동일) |
| 엣지 캐시 | nol.interpark.com | frozen `Date` + `Age` | `Date+Age`로 복구 |

이론적 사각지대(움직이는 live `Date` + 0 아닌 `Age` 동시 송신)는 조사 사이트에 없었고, RFC 비준수다. 그런 서버에서 `Age`가 매초 증가하면 결합값이 2초/초로 뛰어 edge 필터(`900~1100ms`)에 걸려 자동 폴백한다(틀린 값을 확신 있게 내지 않음).

**네이비즘과의 ~10초 차이 진단.** 사용자가 `nol.interpark.com`에서 네이비즘 대비 약 10초 차이를 보고했다. 독립 기준(google live `Date`, RTT 보정)과 동시 비교한 결과 **`Date+Age`는 진짜 UTC와 ±0.5초**, 전체 파이프라인 표시값은 **±0.06초**였다. 즉 frozen `Date`와 실제 시각의 차이(=`Age`값, 그 시점 ≈10초)가 곧 그 격차다. 네이비즘은 CloudFront 캐시 `Date`를 `Age` 보정 없이 읽어 캐시 나이만큼 과거를 표시하는 것으로 보인다 → 이 도구(보정 적용)가 더 정확. 검증 시 `time.is`/휴대폰 NTP 시계와 나란히 비교하면 도구가 벽시계와 일치, 네이비즘이 뒤처짐.

### 4.5 네이버 경로

`naver.com` 호스트는 HTTP Date 대신 네이버 시계 API(`ts-proxy.naver.com/dcontent/util/time.naver`)의 **밀리초 포함** 시각을 쓴다. 초 양자화가 없어 edge detection 불필요 → 저지터 샘플의 offset 중앙값(`naver-time-api`). passportKey는 검색 페이지에서 추출하며 만료 시 재발급한다.

## 5. 정밀도와 오차 모델

```
total_error = date_quantization + rtt_asymmetry + sampling_reduction
            + local_clock_anchor + target_server_clock + browser_delivery
```

PC 시계의 절대 오차는 핵심 항이 아니다(측정 시점 오프셋만 쓰고 이후는 monotonic anchor·`performance.now()` 보간).

교집합은 edge 한 개 σ ≈ `(R+I)/√12`에서 출발해 **1/n**로 좁아진다. 아래 표는 median(1/√N) 기준의 보수적 추정으로, 교집합은 보통 더 정확하다:

| RTT | Count | (median 기준) σ_final |
|---:|---:|---:|
| 50ms | 60 | ±15ms |
| 150ms | 30 | ±25ms |
| 300ms | 17 | ±37ms |

### ⚠️ 교집합 ±의 한계 (중요)

`ci95Ms`(교집합 계열에서 = 폭/2)는 **edge들이 "서로" 일치하는 폭**이지 **정확도 보장이 아니다**. RTT 비대칭이 있으면 모든 edge 구간이 같은 방향으로 평행이동해 상호 일치를 유지한 채 집단 편향을 만든다 → 교집합으로는 못 잡는다. 몬테카를로(800회): 비대칭 0이면 coverage 100%지만, 비대칭 +40/+80ms에서는 `edge-intersect`가 여전히 100% 뜨면서 진짜 θ가 보고 ± 밖에 98~99% 있었다. 그래서 UI는 ✓ 같은 "성공" 표시를 쓰지 않고 "상호 일치(정확도 보장 아님)"로 표기한다.

### 현실적인 기대 정확도

| 환경 | 일반 URL | 네이버 |
|---|---:|---:|
| RTT 안정, Date 경계 정상 | ±30~80ms | ±10~50ms |
| RTT 변동 큼, CDN/프록시 | ±80~200ms | ±50~120ms |
| Date 캐시/비정상 갱신 | 수백 ms+ | API 지연 시 증가 |
| 대상 서버 시계 자체 오차 | 그대로 노출 | 그대로 노출 |

## 6. 재측정 정책

1분 자동 재측정은 없다. 트리거는 앱 시작 초기 측정, 새 URL 입력, F5 새로고침/재진입뿐이다.

- **첫 측정·측정 버튼·새 URL(타겟 변경)**: edge가 적어도 결과를 띄운다(최대 20초).
- **기존 대상 재측정(진짜 재측정)**:
  - `acceptedCount < 5`(edge 부족) → **재측정 실패**. 적은 edge로 갱신하지 않고 기존 offset 유지(`failed-insufficient-edges`).
  - edge 충분 + |Δoffset| ≤ 30ms → 미세 흔들림으로 보고 기존값 유지(`kept-small-delta`).
  - edge 충분 + 30ms < |Δoffset| ≤ 100ms → 반영. 초과 시 1회 더 시도, 2회차도 초과면 기존값 유지(`rejected`).
  - **15초 하드캡**: 2회 시도 전체에 공유되는 15초 예산. fallback 누적으로 20초+ 걸리던 문제 차단. 데드라인 직전(현재 timeout만큼 여유)부터 새 요청을 시작하지 않아 in-flight 1건 포함 15초를 넘지 않는다.

## 7. 표시와 안전마진

UI는 `/api/state`를 1초 폴링하되 매 프레임 서버를 호출하지 않고 `performance.now()`로 로컬 보간한다. 새 기준과 현재 추정 차이가 120ms 초과면 즉시 점프, 이하면 천천히 보정(slew).

"절대 빠르면 안 됨"(수강신청·티켓팅) 보호를 위해 표시 시각을 안전마진만큼 늦춘다. RTT 비대칭(업로드 정체)이 표시를 빠르게 밀 수 있고 그 위험은 RTT에 비례하므로, 고정 30ms가 아니라 **`max(30ms, RTT×0.3)`** 적응형이다(RTT 90→30, 200→60, 400→120ms).

## 8. Clicker 관련 오차

GUI clicker는 `/api/state`로 기준점을 한두 번 잡고 이후 Python `time.perf_counter()`로 목표 시각까지 대기한다. 클릭 정확도엔 Windows 스케줄러 깨움 지연, `SendInput`/브라우저 이벤트 처리 지연, 클릭 후 요청 지연이 추가된다. 기본값 `Lead=0, Resync before=3000, Final spin=25`에서 일관되게 늦으면 `Lead`만 조금씩 올린다.

## 9. 한계와 리스크

- 대상이 `Date`를 안 주면 측정 불가. CDN이 `Date`를 고정하고 `Age`로 경과를 주는 경우는 4.4로 보정하지만, `Age` 없이 `Date`만 재작성/캐시하는 프록시는 여전히 정확도 저하/`upper-envelope` 폴백.
- 로컬 도구라 SSRF 위험은 낮지만 내부망 URL을 넣으면 그 주소로 요청함.
- HTTP keep-alive 재사용 최적화 미구현(RTT 분산에 영향).
- 네이버 API는 비공식이라 형식 변경 시 깨질 수 있음.
- 대규모 실측 로그 분석 자동화 미구현(`logs/probe-YYYYMMDD.jsonl` 누적 분석 여지).
