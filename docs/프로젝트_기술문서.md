# 프로젝트 기술 문서

최종 업데이트: 2026-05-25 +09:00 (KST)

이 문서는 현재 구현(Windows PowerShell 로컬 서버시간 측정 도구)의 구조·API·측정 알고리즘·성능/오차·운영 정책을 한곳에 정리한 기술 레퍼런스다. 사용자용 시각 자료는 `알고리즘_설명.html`(전체 흐름 + 방법1/2 비교)과 `사용설명서.html`에 있다. 인터파크 관련 조사와 회귀 원인은 `인터파크_측정_정리.html`에 따로 정리했다. 과거 변경의 상세 로그는 루트 `CLAUDE.md`를 본다.

> 주의: `CLAUDE.md`에는 과거 NAS/Node.js 장기 설계안이 함께 남아 있다. 현재 동작 판단은 이 문서와 코드(`src/`)를 우선한다.

---

## 1. 개요

사용자가 입력한 웹사이트 URL의 서버 시각을 추정해 브라우저에 실시간 시계로 표시하는 PC용 로컬 도구다. 외부 배포형 서비스가 아니라 `ServerTimeProbe.exe` 또는 `run.bat`로 실행하는 로컬 HTTP 서버 + 정적 웹 UI 구조다. 목표 정확도는 좋은 조건에서 ±50ms 수준이다.

## 2. 실행 흐름과 주요 파일

1. `ServerTimeProbe.exe` 또는 `run.bat` 실행 → `src/probe.ps1`
2. 로컬 HTTP 서버를 `127.0.0.1:8765`에 띄우고 브라우저 자동 오픈
3. 상단 입력창에 URL 입력 → 서버가 적응형(약 6초) 샘플링으로 오프셋 계산
4. 웹 UI가 `/api/state` 기준 시각을 `performance.now()`로 보간해 표시

| 파일 | 역할 |

## 3. API

- `GET /` — 정적 UI. 첫 페이지 로드는 재측정을 걸지 않고, 같은 대상이 설정된 뒤 새로고침/재진입 시 재측정.
- `POST /api/target` — `{ "url": "https://example.com/" }`. 스킴 없으면 `https://` 부착, `http`/`https`만 허용.
- `GET /api/state` — 현재 측정 상태. 주요 필드:

| 필드 | 의미 |
|---|---|
| `host` / `targetUrl` | 현재 표시 대상. 일부 사이트는 canonical 측정 대상으로 정규화됨 |
| `measurementUrl` / `measurementNote` | 실제 HTTP Date를 측정하는 URL / 사이트별 측정 메모 |
| `offsetMs` | PC 기준 대비 서버 시각 오프셋 |
| `rttMedianMs` | 샘플 RTT 중앙값 |
| `sigmaMs` / `ci95Ms` | edge 중점 산포 / 보고 ± (교집합 계열은 일치 폭/2) |
| `method` | `edge-intersect` / `edge-intersect-robust` / `edge-median` / `upper-envelope` / `naver-time-api` |
| `acceptedCount` / `edgeCount` | 채택(합의) edge 수 / 전체 edge 수 |
| `intersectWidthMs` | 교집합 폭(feasible 영역) |
| `status` | `idle` / `queued` / `measuring` / `ok` / `failed` / `stale` |
| `lastRemeasureResult` | `accepted` / `kept-small-delta` / `rejected` / `failed-insufficient-edges` / `fast` |
| `lastMeasureMode` | 마지막 완료 측정 종류: `normal`(정밀) / `fast`(빠른 측정). UI 라벨용 |
| `lastError` | 최근 측정 실패 사유. 기존 측정값이 있으면 실패해도 offset은 유지 |
| `pcSendTimeAtMs` | 응답 직전 PC 기준 epoch ms |
| `ntpInfo` | 참고용 PC-NTP skew |

- `POST /api/remeasure` — 같은 대상 재측정 트리거. `LastMeasureAt`이 없으면 409(먼저 측정 필요). 최대 15초 예산, delta guard 적용(§6).
- `POST /api/measure-fast` — **빠른 측정**(저정확도, 최대 5초) 트리거. `TargetUrl`만 있으면 되고(`LastMeasureAt` 불필요 → 첫 측정으로도 사용 가능), 게이트/재시도 없이 1회 결과를 그대로 채택한다(§4.6, §6).
- `GET /api/samples` — 측정 상세 시각화용. 샘플 배열과 edge 배열(`lowerMs`/`upperMs`/`offsetMs`/`edgePcMs`), `method`, `intersectWidthMs` 등.

## 4. 측정 알고리즘

### 4.1 Cristian + 적응형 다중 샘플(순차)

한 샘플: `t1` 송신, `t2` 수신, `Ts`=Date 헤더 → `offset = Ts + RTT/2 − t2`. 샘플 루프는 **순차**(한 응답 받고 50ms 쉰 뒤 다음 요청)라 1회 비용이 `RTT + 50ms`다. 먼저 3회 RTT 프로브로 median RTT를 잰 뒤 `Count = ceil(6000/(RTT+50))`를 [10,60]로 클램프해 약 6초 윈도우를 유지한다.

병렬 발사를 안 하는 이유: (1) 서버 매너/차단 위험, (2) 동시 요청이 RTT를 혼잡 지연으로 오염시켜 `Ts+RTT/2` 가정이 깨짐, (3) edge detection이 시간순 샘플을 전제로 함.

per-sample timeout도 적응형 `max(1초, ceil(5×RTT))`(정수 초 하한, 캡 5초)으로, hang 1회가 윈도우를 통째로 먹는 것을 막는다.

### 4.2 HEAD vs GET

`HEAD` 우선, 실패 시 `GET + Range: bytes=0-0` 폴백(둘 다 `Date` 동일). HEAD는 일부 서버(구형 PHP/WAF)에서 hang될 수 있어 적응형 timeout으로 방어한다.

### 4.3 Edge detection → 오프셋 추정 (교집합 기본)

`Date` 헤더는 초 단위라 밀리초가 없다. 그래서 `Date`가 `N초→N+1초`로 바뀌는 **경계**를 인접 두 샘플 사이에서 잡는다. 각 edge는 오프셋 θ에 대한 구간 제약 `θ ∈ [S−R, S−L]`을 준다(L,R = 양 끝 서버이벤트 PC시각, S = 새 정수 초).

θ는 측정 내내 상수이고 서버 초 경계는 정확히 1초 간격이므로, **모든 구간의 교집합**이 곧 추정이다(균등노이즈의 MLE, edge 수 n에 대해 1/n 수렴). 모순(outlier)에 대비한 하이브리드:

- 전체 일치 → `edge-intersect`
- 일부 outlier 제외 후 일치(최대 겹침 ≥2) → `edge-intersect-robust`
- 어떤 두 edge도 안 겹침 → `edge-median`(중점들의 median 폴백)
- 경계 자체가 안 보임(캐시/이상) → `upper-envelope`(raw offset 상한봉투)

> 과거에는 "중점들의 median"이 기본이었으나(1/√n), 2026-05-24부터 1초 격자 교집합이 기본이다. 방법1(median) vs 방법2(교집합) 상세 비교는 `알고리즘_설명.html`을 본다.

RTT 필터(`Get-RttThreshold = rttMedian×1.5+2`)와 Gap 필터((rttMedian+50)×3)로 σ가 폭주하는 edge는 제외한다. edge가 부족하면(MinEdgeCount=8 미만) 윈도우를 최대 3회×3초 연장한다(`Test-ShouldExtendWindow`).

### 4.4 CDN 캐시 보정 (`Date + Age`, **조건부**)

CloudFront 같은 엣지 캐시 뒤의 사이트는 `Date`를 **캐시 적재 시각에 고정**하고, 그 뒤 경과 시간을 `Age` 헤더(초, RFC 9111)로 노출할 수 있다. 이때 `Date`만 보면 값이 안 변해 edge가 0개 → `upper-envelope` 폴백 + 정지된 시각이 나온다. 이런 경우에만 실효 서버시각을 `Date + Age`로 복구한다(`Get-EffectiveServerDateMs`).

**그러나 무조건 `Date+Age`를 더하면 안 된다(2026-05-24 회귀 수정).** `Date`가 이미 라이브(매초 증가)인데 `Age`도 0이 아닌 응답(CDN 재검증/edge별 캐시 상태)에선 `Date+Age`가 **정수 초만큼 미래로 과보정**된다. 이 경우 모든 샘플이 같은 `Age`만큼 평행이동하므로 **edge끼리는 완벽히 일치(깔끔한 `edge-intersect`)하면서도 표시가 N초 빨라지는** 위험한 상태가 된다. 티켓팅에서 빠른 표시는 최악이다.

올바른 규칙: **`Age` 보정은 raw `Date`가 윈도우 내내 정지(frozen)일 때만 적용**한다(`Set-AgeCorrectedServerDates`). 핵심은 *라이브 `Date`면 raw `Date`가 이미 정답이라 `Age`가 불필요*하다는 점이다. 판정:

- raw `Date`의 span(최대−최소) ≤ 2000ms **그리고** `Age>0` 존재 → **frozen 캐시**로 보고 `ServerDateMs = Date + Age*1000` (`AgeCorrected=true`).
- 그 외(raw `Date`가 윈도우만큼 증가 = 라이브) → **raw `Date` 사용, `Age` 무시**(`AgeCorrected=false`). → 절대 과보정(미래) 없음.

`Invoke-HeadProbe`는 `RawServerDateMs`/`AgeSec`를 따로 들고 오고, 기본 `ServerDateMs`는 raw(안전). `Reduce-Samples`가 위 판정으로 frozen일 때만 결합한다. 결과에 `AgeCorrected` 필드 노출.

> 실측 검증(2026-05-24/25, 같은 PC): `ticket.interpark.com`과 `nol.interpark.com/ticket`은 현재 티켓 서비스 기준으로 `https://nol.interpark.com/ticket?t=...`를 재는 쪽이 네이비즘과 가장 잘 맞았다. 이 경로는 최근 관측에서 `X-Cache: Miss from cloudfront`, `Age` 없음, `Cache-Control: no-store`가 자주 보였고 raw `Date`가 정상 edge를 제공했다. 인터파크 전용 상세는 `인터파크_측정_정리.html` 참고.

**범용성 검증(2026-05-24).** 7개 사이트 헤더 조사 결과 두 부류로만 갈렸고, 둘 다 올바르게 처리된다:

| 부류 | 사이트 | 거동 | 이 보정의 효과 |
|---|---|---|---|
| 일반 origin | google, github, cloudflare, interpark.com, ticket.interpark.com, melon티켓 | live `Date`, `Age` 없음 | `Age=0` → 무변화(기존과 동일) |
| 엣지 캐시 | 일부 CloudFront/프록시 응답 | frozen `Date` + `Age` | `Date+Age`로 복구 |

위 "live `Date` + 0 아닌 `Age` 동시 송신"이 바로 §4.4의 조건부 규칙이 막는 케이스다. 무조건 더하던 v1은 여기서 정수 초 과보정을 일으켰다(아래 회귀 참고).

**네이비즘과의 초 단위 차이 진단(정정 포함).**
1. `ticket.interpark.com`과 `nol.interpark.com`은 네이비즘에서 같은 서비스 기준처럼 보인다. 원본 도메인 하나만 raw로 재는 방식보다 최종 티켓 페이지 경로(`nol.interpark.com/ticket`)를 canonical 측정 대상으로 삼는 것이 더 안정적이었다.
2. +5~6초 빠르게 보인 핵심 회귀는 서버 offset 자체가 아니라 **브라우저 UI 기준점 재동기화 실패**였다. `/api/state`는 정상 offset을 주는데 `clock.js`가 이전 `baseServerMs`를 같은 `lastMeasureAt`이라는 이유로 계속 사용했고, F5를 누르면 JS 메모리가 초기화되어 정상으로 보였다. 현재는 `measurementUrl`/요청시각 변경 및 화면 추정치와 state 기준 시각의 500ms 이상 괴리 시 강제 재기준화한다.
3. 수동 재측정은 같은 canonical `targetUrl`이면 기존 offset 보호 정책을 적용한다. 기존값과 30ms 이하는 기존 유지, 31~100ms는 반영, 100ms 초과는 1회 재시도 후에도 초과하면 기존 유지.

교훈: 라이브 `Date`면 `Age`는 불필요·유해. `Age`는 `Date`가 정지일 때만 의미가 있다.

### 4.5 네이버 경로

`naver.com` 호스트는 HTTP Date 대신 네이버 시계 API(`ts-proxy.naver.com/dcontent/util/time.naver`)의 **밀리초 포함** 시각을 쓴다. 초 양자화가 없어 edge detection 불필요 → 저지터 샘플의 offset 중앙값(`naver-time-api`). passportKey는 검색 페이지에서 추출하며 만료 시 재발급한다.

### 4.6 빠른 측정 (저정확도, 최대 5초)

"20초를 못 기다리는 사용자"를 위한 **별도 1회성** 측정이다(`POST /api/measure-fast`, UI `⚡ 빠른 측정` 버튼). 신규 측정 함수 없이 기존 `Invoke-AdaptiveMultiSample`을 빠른 파라미터로 호출한다:

```
Invoke-AdaptiveMultiSample -TargetWindowMs 2500 -MinEdgeCount 1 -MaxTotalMs 5000 -DefaultTimeoutSec 2
```

- `TargetWindowMs 2500`: 메인 윈도우 ~2.5초 → edge 1~3개 기대.
- `MinEdgeCount 1`: 윈도우 연장 루프를 사실상 비활성화(edge ≥1이면 연장 안 함, 0이면 `upper-envelope`라 연장 대상 아님) → 5초 내 종료.
- `MaxTotalMs 5000`: 하드캡 5초.
- **`DefaultTimeoutSec 2`(필수)**: RTT 프로브 단계의 데드라인 reserve가 `DefaultTimeoutSec`다. 기본 5초면 `MaxTotalMs=5000`과 같아져 `elapsed(0)+5000 ≥ 5000`이 즉시 참이 되고, 첫 프로브도 못 돌린 채 `All initial RTT probes failed`로 죽는다. 2초로 낮춰 5초 예산 안에서 프로브가 실행되게 한다(적응형 timeout도 ≤2초로 묶여 cap 유지).

**수용 정책**: 빠른 측정은 명시적 동작이라 재측정 게이트(edge<5 실패)·delta 재시도를 **타지 않고** 1회 결과를 그대로 채택·덮어쓴다. 정밀 offset이 이미 있어도 덮어쓰므로, **클라이언트가 최근(5분 이내, stale 판정 재사용) 측정이 있으면 `confirm()`로 확인**한 뒤에만 요청을 보낸다. 오래됨/없음이면 바로 실행. 서버는 들어온 요청은 무조건 실행한다(확인 책임은 클라이언트).

빠른 측정 후에는 `LastMeasureMode`가 다시 `normal`이 되므로, 이어지는 F5 재측정은 기존 15초 정밀 경로를 탄다. 결과는 UI에서 `⚡ 빠른 측정 · … (정확도 낮음)`으로 표기한다. 기대 정확도: edge 1~3개 → ±100~300ms, edge 0개(frozen Date) → ±500ms.

> 실측(google, 2026-05-25): 초기측정 `edge-intersect` 10개/10초 → 빠른 측정 `edge-intersect` 3개/3.3초, offset이 정밀값과 ~20ms 차. 5초 cap 내 동작 확인.

## 5. 정밀도와 오차 모델

```
total_error = date_quantization + rtt_asymmetry + sampling_reduction
            + local_clock_anchor + target_server_clock + browser_delivery
```

PC 시계의 절대 오차는 핵심 항이 아니다(측정 시점 오프셋만 쓰고 이후는 monotonic anchor·`performance.now()` 보간).

교집합은 edge 한 개 σ ≈ `(R+I)/√12`에서 출발해 **1/n**로 좁아진다. 아래 표는 median(1/√N) 기준의 보수적 추정으로, 교집합은 보통 더 정확하다:

| RTT | Count | (median 기준) σ_final |
|---:|---:|---:|
| 50ms | 60 | ±15ms |
| 150ms | 30 | ±25ms |
| 300ms | 17 | ±37ms |

### ⚠️ 교집합 ±의 한계 (중요)

`ci95Ms`(교집합 계열에서 = 폭/2)는 **edge들이 "서로" 일치하는 폭**이지 **정확도 보장이 아니다**. RTT 비대칭이 있으면 모든 edge 구간이 같은 방향으로 평행이동해 상호 일치를 유지한 채 집단 편향을 만든다 → 교집합으로는 못 잡는다. 몬테카를로(800회): 비대칭 0이면 coverage 100%지만, 비대칭 +40/+80ms에서는 `edge-intersect`가 여전히 100% 뜨면서 진짜 θ가 보고 ± 밖에 98~99% 있었다. 그래서 UI는 ✓ 같은 "성공" 표시를 쓰지 않고 "상호 일치(정확도 보장 아님)"로 표기한다.

### 현실적인 기대 정확도

| 환경 | 일반 URL | 네이버 |
|---|---:|---:|
| RTT 안정, Date 경계 정상 | ±30~80ms | ±10~50ms |
| RTT 변동 큼, CDN/프록시 | ±80~200ms | ±50~120ms |
| Date 캐시/비정상 갱신 | 수백 ms+ | API 지연 시 증가 |
| 대상 서버 시계 자체 오차 | 그대로 노출 | 그대로 노출 |

## 6. 재측정 정책

1분 자동 재측정은 없다. 트리거는 앱 시작 초기 측정, 새 URL 입력, 재측정 버튼, 빠른 측정 버튼, F5 새로고침/재진입뿐이다.

측정 종류별 하드캡: **초기/타겟 변경 20초 · 재측정 15초 · 빠른 측정 5초**(UI 안내줄에 명시). 캡은 *목표가 아니라 상한*이라, 충분한 edge에 일찍 도달하면 그 전에 끝난다.

- **첫 측정·측정 버튼·새 URL(타겟 변경)**: 새 측정값을 반영한다(최대 20초). 같은 URL을 다시 입력해도 새 측정으로 취급한다.
- **재측정 버튼**: 같은 canonical `targetUrl`이면 `PendingTargetChange=false`로 두고 기존 offset과 새 offset의 delta guard를 적용한다(최대 15초, 2회 시도 공유 예산). 측정 경로가 캐시버스터를 쓰는 사이트는 매번 새 `measurementUrl`을 만들지만, 이것만으로는 타겟 변경이 아니다.
- **재측정 반영 기준**: edge 5개 미만은 `failed-insufficient-edges`로 기존값 유지, delta 30ms 이하는 `kept-small-delta`로 기존값 유지, 31~100ms는 `accepted`, 100ms 초과는 1회 더 측정하고 두 번째도 초과하면 `rejected`로 기존값 유지.
  > 재측정 예산을 10초→**15초**로 올린 이유(2026-05-25): 로그상 `failed-insufficient-edges`의 9/10이 `edge-intersect`(전부 상호 일치)인데 edge가 4개로 1개 부족한 "clean but short" 케이스였다. 같은 호스트가 초기측정(20초)에선 edge 9개, 재측정(10초)에선 4개 → 순수 시간 부족. 캡은 상한이라 빠른 호스트엔 비용 0, 느린 호스트만 5개 도달할 시간을 더 받는다. edge 게이트(5)는 정확도/안전 보장이라 그대로 유지.
- **빠른 측정 버튼**: 위 게이트·delta·재시도를 **모두 우회**하고 1회 결과를 그대로 채택한다(저정확도). 상세는 §4.6. 정밀 offset 덮어쓰기 보호는 클라이언트 `confirm()`뿐이다.
- **재측정 실패**: 기존 `lastMeasureAt`이 있으면 화면을 `failed`로 날리지 않고 기존 offset을 유지하며 `lastError`/상태 문구만 갱신한다.

## 7. 표시와 안전마진

UI는 `/api/state`를 1초 폴링하되 매 프레임 서버를 호출하지 않고 `performance.now()`로 로컬 보간한다. 새 기준과 현재 추정 차이가 120ms 초과면 즉시 점프, 이하면 천천히 보정(slew).

표시 동기화 규칙:

- 새 `targetUrl`, 새 `measurementUrl`, 새 `lastMeasureRequestedAt`이 오면 기존 `baseServerMs`를 비운다.
- 새 `lastMeasureAt`이 오면 `/api/state` 기준으로 다시 `setClockBase()`를 적용한다.
- 같은 `lastMeasureAt`이라도 화면 추정치와 `/api/state` 기준 시각이 500ms 이상 벌어지면 강제 재기준화한다. 이 규칙은 “F5하면 정상, 버튼만 누르면 +5~6초 유지” 회귀를 막기 위한 것이다.
- `queued`는 내부 타이머 대기 상태다. UI에는 수동 재측정이면 `재측정 중...`, 새 대상이면 `초기 측정 중...`으로 표시한다.

"절대 빠르면 안 됨"(수강신청·티켓팅) 보호를 위해 표시 시각을 안전마진만큼 늦춘다. RTT 비대칭(업로드 정체)이 표시를 빠르게 밀 수 있고 그 위험은 RTT에 비례하므로, 고정 30ms가 아니라 **`max(30ms, RTT×0.3)`** 적응형이다(RTT 90→30, 200→60, 400→120ms).

## 8. Clicker 관련 오차

GUI clicker는 `/api/state`로 기준점을 한두 번 잡고 이후 Python `time.perf_counter()`로 목표 시각까지 대기한다. 클릭 정확도엔 Windows 스케줄러 깨움 지연, `SendInput`/브라우저 이벤트 처리 지연, 클릭 후 요청 지연이 추가된다. 기본값 `Lead=0, Resync before=3000, Final spin=25`에서 일관되게 늦으면 `Lead`만 조금씩 올린다.

## 9. 한계와 리스크

- 대상이 `Date`를 안 주면 측정 불가. CDN이 `Date`를 고정하고 `Age`로 경과를 주는 경우는 4.4로 보정하지만, `Age` 없이 `Date`만 재작성/캐시하는 프록시는 여전히 정확도 저하/`upper-envelope` 폴백.
- 로컬 도구라 SSRF 위험은 낮지만 내부망 URL을 넣으면 그 주소로 요청함.
- HTTP keep-alive 재사용 최적화 미구현(RTT 분산에 영향).
- 네이버 API는 비공식이라 형식 변경 시 깨질 수 있음.
- 대규모 실측 로그 분석 자동화 미구현(`logs/probe-YYYYMMDD.jsonl` 누적 분석 여지).

## 10. 변경 이력 (요약)

상세는 `CLAUDE.md` 참고.

- **2026-05-02 위상지연 수정**: `Date`에 일괄 `+500ms` 더하던 방식을 raw Cristian + edge transition 기반으로 교체. 1분 자동 재측정 제거, F5 재측정 도입. clock.js slew 보정 추가.
- **2026-05-08 적응형 샘플링**: 고정 50샘플/100ms → `IntervalMs=50` 고정 + `Count` 적응형(약 6초 윈도우) + 적응형 timeout + 윈도우 연장.
- **2026-05-24 교집합 + 재측정 정책 + 안전마진**: edge 오프셋 추정을 median → 1초 격자 교집합(하이브리드)으로 전환. `Get-Median` 홀수 n 반올림 버그 수정. 재측정 edge<5 실패 + 하드캡(당시 10초, 후속 15초로 상향). 적응형 안전마진. 교집합 과신 audit 반영(라벨 정직화).
- **2026-05-24 CDN `Age` 보정**: CloudFront 등 frozen-`Date` 캐시(예: `nol.interpark.com`)에서 `upper-envelope` 폴백 + 시각 정지 문제 수정. 실효 서버시각을 `Date + Age`로 계산(`Get-EffectiveServerDateMs`) → edge 교집합 복구(§4.4).
- **2026-05-25 인터파크 canonical 측정/표시 회귀 수정**: `ticket.interpark.com`, `tickets.interpark.com`, `nol.interpark.com`을 `https://nol.interpark.com/ticket`으로 정규화하고 실제 측정은 `?t=` 캐시버스터가 붙은 `measurementUrl`로 수행. 같은 canonical target 재측정은 캐시버스터가 바뀌어도 delta guard를 적용. `clock.js`가 낡은 `baseServerMs`를 유지해 F5 전까지 +5~6초 빠르게 표시하던 회귀를 `measurementUrl`/요청시각 추적과 500ms drift 강제 재기준화로 수정.
- **2026-05-25 재측정 캡 10→15초**: 로그 분석 결과 `failed-insufficient-edges`가 "clean but short"(edge-intersect 4개, 1개 부족)였고 같은 호스트가 20초 예산엔 edge 9개를 뽑음. 재측정 예산을 15초로 상향(§6). 캡은 상한이라 빠른 호스트 비용 0.
- **2026-05-25 빠른 측정 추가**: 최대 5초 저정확도 1회성 측정(§4.6). `POST /api/measure-fast`, state `MeasureMode`/`LastMeasureMode`, UI `⚡ 빠른 측정` 버튼 + 최근 측정 시 `confirm()` 덮어쓰기 확인 + cap 3개 안내. 기존 `Invoke-AdaptiveMultiSample` 재사용(신규 함수 0개), `DefaultTimeoutSec=2`로 5초 budget 내 프로브 실행 보장. 설계/계획은 `docs/superpowers/specs|plans/2026-05-25-fast-measurement*`.
