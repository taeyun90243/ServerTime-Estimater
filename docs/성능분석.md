# 성능분석

최종 업데이트: 2026-05-08 +09:00 (KST)

이 문서는 현재 구현된 Windows PowerShell 로컬 서버시간 측정 도구의 성능 특성과 오차 요인을 정리한다. 이전 문서의 NAS/Node.js 장기 설계 가정이 아니라, 현재 코드(`src/probe.ps1`, `src/measurement.ps1`, `src/http-server.ps1`, `src/web/*`) 기준이다.

## 현재 측정 방식

일반 URL은 HTTP 응답의 `Date` 헤더를 적응형 샘플링(약 6초 윈도우)으로 수집해 대상 서버 시각과 PC 시각의 오프셋을 추정한다. `HEAD` 요청을 우선 사용하고, 실패하면 `GET` + `Range: bytes=0-0`로 폴백한다.

네이버(`https://naver.com/`)는 특수 경로로 처리한다. 일반 `Date` 헤더 대신 네이버 시계 API(`ts-proxy.naver.com/dcontent/util/time.naver`)가 반환하는 밀리초 포함 시각을 사용한다.

측정 결과는 로컬 HTTP 서버(`127.0.0.1:8765`)가 웹 UI에 `/api/state`로 전달한다. 웹 UI는 한 번 받은 기준 시각에 `performance.now()` 증가분을 더해 화면을 계속 갱신한다.

## 오차 모델

추정 오차는 아래 항들의 합으로 볼 수 있다.

```
total_error =
  date_quantization_error
  + rtt_asymmetry_error
  + sampling_reduction_error
  + local_clock_anchor_error
  + target_server_clock_error
  + browser_delivery_error
```

PC 시계의 절대 오차는 핵심 오차 항이 아니다. 현재 구현은 측정 시점의 PC 기준과 대상 서버 기준 사이의 오프셋을 구한 뒤, 이후 진행은 `Stopwatch` 기반 monotonic anchor와 브라우저 `performance.now()`로 보간한다. 따라서 PC 시계가 표준시와 몇 초 어긋나 있어도 같은 프로세스 안에서의 표시값에는 직접 반영되지 않는다.

## 일반 URL 정확도

일반 URL은 HTTP `Date` 헤더가 초 단위라는 점이 가장 큰 제약이다. 단일 샘플에 `+500ms`를 더하는 단순 보정은 평균 편향은 줄이지만 초침 위상 지연을 만들 수 있어 현재 기본 방식이 아니다.

현재 구현은 다음 순서로 줄인다.

1. 먼저 3회 RTT 프로브로 median RTT를 측정한다.
2. `N = ceil(6000 / (RTT + 50))`을 계산해 [10, 60] 범위로 클램프한다 (총 윈도우 약 6초).
3. 남은 샘플을 50ms 간격으로 채우고 각 샘플의 `Date` 값, RTT, PC 수신 시각을 기록한다.
4. `Date` 헤더가 `N초 -> N+1초`로 증가하는 경계를 찾는다.
5. 경계 전후 샘플의 서버 이벤트 추정 시점을 이용해 초 경계 오프셋 후보를 만든다.
6. 후보 중앙값을 최종 오프셋으로 사용한다.
7. 경계가 전혀 보이지 않는 비정상/캐시 상황에서는 raw offset upper-envelope 방식으로 폴백한다.

### 샘플링 파라미터 최적화 (2026-05-08)

샘플 1회 비용은 `RTT + IntervalMs`이고 총 시간은 `Count × (RTT + IntervalMs)`다. 같은 윈도우 안 기대 edge 수는 `윈도우/1000ms`다.

Edge 1개당 표준편차 `σ_edge ≈ (R+I)/√12`, 후보 median의 표준오차는 `σ_final ≈ 0.362 × √(1000(R+I)/N)`. 따라서:

- `IntervalMs`는 작을수록 시간·정밀도 양쪽에 이득. 서버 매너상 하한 50ms를 채택.
- 안정적 median에는 edge 5~6개 필요 → 윈도우 약 6초가 시간 하한.
- 결론: `IntervalMs=50` 고정, `Count`는 RTT에 따라 적응시켜 6초 윈도우 유지.

기대 정밀도(이론치):

| RTT | Count | σ_final |
|---:|---:|---:|
| 50ms | 60 | ±15ms |
| 150ms | 30 | ±25ms |
| 300ms | 17 | ±37ms |

실측은 RTT 필터 탈락·CDN 캐시·jitter 영향으로 위 값의 1.5배 정도로 본다.

현실적인 기대 정확도:

| 환경 | 기대 수준 |
|---|---:|
| 같은 국내망, RTT 안정, `Date` 경계 정상 | 대략 ±30~80ms |
| RTT 변동 큼, CDN/프록시 개입 | 대략 ±80~200ms |
| `Date` 헤더 캐시/비정상 갱신 | 수백 ms 이상 가능 |
| 대상 서버 시계 자체가 틀림 | 그 서버의 틀린 시각을 그대로 표시 |

이 값은 모델 기반 추정이다. 현재 저장소에는 대규모 실측 데이터셋이 없으므로 `logs/probe-YYYYMMDD.jsonl`을 누적해서 사이트별 분포를 다시 계산해야 더 정확해진다.

## 네이버 URL 정확도

네이버는 일반 `Date` 헤더보다 유리하다. 네이버 시계 API 응답에 밀리초가 포함되어 있어 초 단위 양자화 오차가 거의 사라진다.

현재 네이버 경로는 다음 방식이다.

1. 적응형 샘플링(약 6초 윈도우, 위와 동일 공식)으로 네이버 시계 API를 호출한다.
2. 응답의 `yyyy/MM/dd/HH/mm/ss/SSS` 값을 KST에서 UTC epoch ms로 변환한다.
3. RTT/2를 보정한다.
4. RTT median 기준 저지터 샘플만 남긴 뒤 offset 중앙값을 사용한다.

현실적인 기대 정확도:

| 환경 | 기대 수준 |
|---|---:|
| RTT 안정 | 대략 ±10~50ms |
| RTT 변동 큼 | 대략 ±50~120ms |
| API 응답 지연/캐시 이상 | 더 커질 수 있음 |

## 재측정 정책

현재 구현은 1분 자동 재측정을 하지 않는다.

- 앱 시작 시 `-TargetUrl`이 있으면 초기 적응형 측정
- 기본 실행(`run.bat`)은 브라우저에서 사용자가 URL을 입력한 뒤 측정
- 웹 UI에서 새 URL을 입력하면 새 대상 기준으로 적응형 측정
- F5 새로고침 또는 페이지 재진입 시 기존 대상에 대해 적응형 재측정
- 기존 대상 재측정 결과가 기존 offset과 100ms 이내면 반영
- 100ms 초과면 한 번 더 적응형 재측정
- 2회차도 100ms 초과면 기존 offset 유지

새 URL 입력으로 대상이 바뀐 경우에는 기존 offset과 비교하지 않고 새 측정값을 반영한다. 서로 다른 서버 사이의 offset 차이는 정상적인 값이기 때문이다.

## 표시 성능

웹 UI는 `/api/state`를 1초마다 폴링하지만, 매 프레임마다 서버를 호출하지 않는다. 화면 시간은 브라우저의 `performance.now()`로 로컬 보간한다.

작은 측정 흔들림은 바로 점프시키지 않는다. 새 기준과 현재 표시 추정값 차이가 120ms 이하면 천천히 보정(slew)하고, 120ms를 넘으면 즉시 기준을 바꾼다.

## 병목과 리스크

- 대상 서버가 `Date` 헤더를 보내지 않으면 측정 실패
- 일부 서버는 `HEAD`를 막거나 `Date` 헤더를 프록시/CDN에서 재작성할 수 있음
- 로컬 도구라 SSRF 위험은 낮지만, 사용자가 내부망 URL을 넣으면 그 주소로 요청을 보냄
- 적응형 샘플링이라 측정 완료까지 약 6초 필요 (RTT가 매우 크면 더 길어질 수 있음)
- 네이버 특수 API는 공식 공개 계약 API가 아니므로 형식이 바뀌면 깨질 수 있음
- 현재 HTTP 연결 keep-alive 재사용 최적화는 구현되어 있지 않음

## Clicker 관련 오차

GUI clicker는 서버시간 측정 도구의 `/api/state`를 한두 번 읽어 기준점을 잡고, 이후에는 Python `time.perf_counter()`로 목표 시각까지 대기한다. 카운트다운 표시가 네트워크 요청을 계속 발생시키지는 않는다.

클릭 정확도에는 아래 항목이 추가로 영향을 준다.

- Windows 스케줄러가 프로세스를 깨우는 지연
- `SendInput` 호출 및 대상 브라우저의 이벤트 처리 지연
- 클릭 후 웹사이트가 서버에 요청을 보내는 지연

처음에는 `Lead ms=0`, `Resync before ms=3000`, `Final spin ms=25`를 사용한다. 실제 환경에서 클릭이 일관되게 늦으면 `Lead ms`만 조금씩 올려 보정한다.
