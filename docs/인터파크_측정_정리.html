인터파크 서버시간 측정 정리

최종 업데이트: 2026-05-25 KST. 대상: ticket.interpark.com, tickets.interpark.com, nol.interpark.com.

1. 결론

인터파크 티켓 관련 입력은 모두 같은 canonical 측정 대상으로 정규화한다.

입력: ticket.interpark.com
입력: tickets.interpark.com
입력: nol.interpark.com

표시/대상 URL: https://nol.interpark.com/ticket
실제 측정 URL: https://nol.interpark.com/ticket?t=<random>

?t=...는 CloudFront/브라우저/중간 캐시 영향을 줄이기 위한 캐시버스터다. 네이비즘에서도 ticket.interpark.com과 nol.interpark.com이 사실상 같은 서비스 시간처럼 보였기 때문에, 원본 입력 문자열보다 최종 티켓 페이지 경로를 기준으로 삼는다.

2. 왜 이렇게 하는가

관찰의미
ticket.interpark.com은 실제 서비스 흐름에서 nol.interpark.com/ticket 쪽으로 이어진다.입력 도메인 raw Date만 보면 네이비즘/실서비스 흐름과 달라질 수 있다.
네이비즘에서 ticket.interpark.com과 nol.interpark.com이 거의 같은 값으로 표시된다.네이비즘도 개별 raw 도메인보다 서비스 단위/최종 경로 기준에 가깝게 보인다.
nol.interpark.com/ticket?t=... 최근 응답은 X-Cache: Miss, Age 없음, Cache-Control: no-store가 자주 보인다.이 경우 Date + Age가 아니라 raw Date 기준으로 edge를 잡는 것이 맞다.

3. Age 보정 규칙

인터파크에서 한동안 혼선이 있었던 부분이다. 규칙은 단순하다.

X-Cache: Miss + Age 없음
→ Date 그대로 사용

Date가 매초 움직임
→ Date 그대로 사용, Age가 있어도 더하지 않음

Date가 측정 윈도우 내내 멈춤 + Age>0
→ frozen cache로 보고 Date + Age 사용
주의. 라이브 Date에 Age를 무조건 더하면 샘플 전체가 정수 초만큼 미래로 평행이동한다. 이때 edge는 서로 잘 맞아 보여도 화면은 몇 초 빠를 수 있다.

4. +5~6초 빠르게 보였던 실제 원인

마지막으로 확인된 핵심 원인은 서버 측정값 자체가 아니라 브라우저 UI 기준점이었다.

  1. 서버 /api/state는 정상 offset을 내려줬다.
  2. 브라우저 clock.js는 한 번 잡은 baseServerMs를 performance.now()로 계속 굴린다.
  3. 이전 잘못된 기준점이 잡힌 상태에서 같은 lastMeasureAt이면 새 state를 다시 적용하지 않았다.
  4. 그래서 F5를 누르면 JS 메모리가 초기화되어 정상으로 보였고, 버튼만 누르면 오래된 +5~6초 기준점이 남았다.

수정 사항:

  • targetUrl, measurementUrl, lastMeasureRequestedAt 변경 시 기존 baseServerMs 리셋
  • 같은 lastMeasureAt이라도 화면 추정치와 /api/state 기준 시각이 500ms 이상 다르면 강제 재기준화
  • 재측정 버튼을 누르면 즉시 기존 UI 기준점 리셋
  • 수동 재측정은 delta 보호 로직으로 새 값을 버리지 않고 명시적 새 측정으로 반영

5. 현재 기대 동작

동작기대 결과
nol.interpark.com 입력 후 측정입력창/상태가 https://nol.interpark.com/ticket 기준으로 바뀌고 새 ?t=... 경로를 측정한다.
재측정 버튼 클릭UI는 재측정 중...으로 표시하고, 새 measurementUrl로 샘플링한다.
재측정 성공기존 값과 차이가 커도 수동 요청이므로 새 offset을 반영한다.
재측정 실패기존 offset은 유지하고 lastError에 실패 사유를 남긴다.
F5 없이 기다림1초 폴링 중 화면 기준과 서버 state가 500ms 이상 다르면 자동으로 재동기화한다.

6. 다른 사이트에 적용할 원칙

대형 티켓/신청 사이트는 입력 도메인, 리다이렉트 최종 URL, CDN 응답, 내부 API 기준이 다를 수 있다. 사이트별로 아래 후보를 비교하고 필요할 때 canonical 매핑을 추가한다.

1. 입력 URL raw Date
2. 리다이렉트 최종 200 OK Date
3. 캐시버스터를 붙인 최종 URL Date
4. 서비스별 알려진 페이지/대기열/API 경로 Date
5. 네이비즘 HTML에 박힌 thisTime2/mgap 결과와의 차이
