1. 왜 이 방법인가

대상 웹사이트의 서버가 지금 몇 시인지 알고 싶다. 그러나 외부에서 직접 서버 시계를 읽을 수단은 없다. HTTP 응답 헤더의 Date 필드(RFC 9110)가 거의 유일한 단서다.

두 가지 어려움:

  • 해상도: Date는 초 단위. 밀리초가 없다.
  • 지연: 응답을 받았을 때는 이미 그 Date가 찍힌 순간으로부터 RTT 절반만큼 지났다.

이 도구는 이 두 가지를 다음 기법으로 해결한다.

  1. Cristian의 알고리즘으로 RTT/2 보정 (지연 처리)
  2. 여러 샘플의 Date 헤더가 N→N+1로 바뀌는 경계 순간을 찾아 ms 정밀도 회복 (해상도 처리)
  3. 중앙값과 RTT 필터로 outlier 제거 (안정성)
  4. 적응형 샘플 수로 측정 시간/정밀도 균형 (실용성)

2. 기본 원리: Cristian's Algorithm

한 번의 HTTP 요청에서 알 수 있는 것:

  • t1: 요청 보낸 PC 시각
  • t2: 응답 받은 PC 시각
  • Ts: 응답 헤더의 Date 값 (서버가 응답 만든 순간의 서버 시각)
  • RTT = t2 - t1

왕복 시간이 대칭이라고 가정하면(송신 ≈ 수신 지연), 서버가 응답을 만든 순간은 PC 시계로 t2 - RTT/2다. 그 순간 서버 시각이 Ts였으므로:

offset = Ts - (t2 - RTT/2) = Ts + RTT/2 - t2

이후 PC 시각에 offset을 더하면 추정 서버 시각이 나온다.

가정의 한계. RTT 비대칭(예: 송신 5ms, 수신 50ms)이면 위 식은 22ms 어긋난다. CDN/프록시가 응답 도중에 헤더를 재작성하면 Ts 자체가 의미 다른 값이 된다. 이 도구는 비대칭을 보정하지 않는다 — 통계적으로 평균화될 뿐이다.

3. 샘플링: 적응형 다중 샘플

3.1 한 번으로는 안 됨

Date가 초 단위라 단일 샘플로는 ±500ms 양자화 오차가 있다. 또 RTT가 들쭉날쭉하면 단일 offset 추정이 크게 틀어진다. 여러 번 측정 후 통계 처리가 필수.

3.2 순차 샘플링 (병렬 아님)

샘플 루프는 한 응답을 받은 다음 50ms 쉬고 다음 요청이다. 한 사이클 비용 = RTT + 50ms.

요청 → 응답 대기 → 응답 수신 → 50ms sleep → 다음 요청
<-- RTT -->             <-50->

병렬로 50ms마다 발사하지 않는 이유 3가지:

  1. 서버 매너: 한 호스트에 초당 20 in-flight 요청은 rate-limit/차단 위험.
  2. 혼잡 지연: 동시 요청이 같은 큐를 공유하면 측정 RTT가 진짜 네트워크 RTT가 아니라 혼잡 지연이 된다. Cristian의 Ts + RTT/2 가정이 깨짐.
  3. Edge 가정: Edge detection은 인접 두 샘플 사이에 Date 초 전환이 있다는 시간순 가정에 의존. 병렬 발사면 응답 도착 순서가 발신 순서와 어긋날 수 있어 무효.

3.3 적응형 Count

고정 50샘플은 RTT가 큰 사이트에서 시간이 오래 걸리고, 작은 사이트에서는 불필요하게 많다. 대신 6초 윈도우를 목표로 RTT에 맞춰 Count를 자동 조정.

Count = ceil(6000 / (RTT + 50)), clamp [10, 60]

초기 3샘플로 RTT median을 먼저 추정한 뒤 Count 결정. 결과:

RTTCount측정 시간
50ms60 (clamp)~6.0s
100ms40~6.0s
200ms24~6.0s
500ms11~6.0s
1000ms10 (clamp)~10.5s

3.4 적응형 Timeout

한 샘플당 timeout도 RTT에 맞춰 줄인다. 옛날 코드는 5초 고정이라 hung 한 번에 측정의 반이 날아갔다.

timeoutMs = max(5 × RTT, RTT + 300ms, 400ms), cap 3000ms

HTTP Date probe는 HttpWebRequest의 ms 단위 timeout을 직접 써서 정수 초 하한을 피한다.

4. HEAD vs GET

한 샘플은 대상 서버에 HTTP 요청을 보낸다. 메서드 선택지:

항목HEADGET 전체GET + Range bytes=0-0
본문 다운로드0 바이트전체1 바이트
Date 헤더받음받음받음
서버 부담최소큼거의 최소
미지원 서버구형 PHP, 일부 WAF거의 없음거의 없음
hang/timeout 위험가끔드묾드묾

HEAD가 hang되는 이유:

  • HEAD는 실사용 트래픽이 적어 서버측 처리 코드에 버그가 남기 쉽다.
  • 많은 PHP 프레임워크가 HEAD에서도 GET 본문 생성 로직을 끝까지 돌린 뒤 본문만 잘라낸다 → 데드락 발생 가능.

현재 구현은 HEAD 우선, 실패 시 GET + Range: bytes=0-0로 폴백. 둘 다 Date 헤더는 동일하게 받음.

단, HEAD가 실패하고 Range가 성공한 서버는 probeMode=range로 전환한다. 이 판정은 실행 중인 프로세스에서 호스트별로 캐시되며, 이후 샘플과 다음 재측정은 Range 전용으로 시작한다. 그래서 HEAD timeout → Range retry가 매번 반복되어 실패 1회가 timeout 2번으로 늘어나는 상황을 피한다.

5. Edge Detection

5.1 핵심 아이디어

Date 헤더는 초 단위라 직접 ms 정밀도를 못 준다. 하지만 인접 두 샘플의 Date 값이 N → N+1로 바뀌는 순간은 ms 단위 사건이다. 그 순간을 PC 시계에 매핑하면 ms 정밀도 offset이 나온다.

5.2 단계별 추정

1단계 — 각 샘플의 "서버 응답 생성 순간"을 PC 시계로 환산

prev 생성 PC 시각 = prev.pcAtT2 - prev.RTT / 2
curr 생성 PC 시각 = curr.pcAtT2 - curr.RTT / 2

2단계 — Edge가 그 두 시점 사이 어딘가에 있다

prev.Date = 12:00:00, curr.Date = 12:00:01이면 서버 시계의 12:00:01.000 경계는 두 PC 시각 사이 어딘가다. 사전 정보 없으므로 균등분포로 가정해서 중점을 점 추정값으로:

edgePcMs = (prev 생성 PC 시각 + curr 생성 PC 시각) / 2

3단계 — 그 PC 시각에 서버 시각은 curr.Date

offset = curr.Date - edgePcMs

이게 한 edge가 만드는 offset 추정치 하나.

5.3 균등분포 가정의 정밀도

균등분포 σ = 구간폭 / √12. 구간폭 = curr 생성 시각 - prev 생성 시각 ≈ RTT + 50ms.

σ_edge ≈ (RTT + 50) / √12
RTT구간폭σ_edge±2σ
50ms100ms29ms±58ms
100ms150ms43ms±86ms
200ms250ms72ms±144ms
500ms550ms159ms±318ms

한 edge로는 부족. 여러 개 모아 median 취하면 √N로 좁아짐.

6. 필터링: RTT와 Gap

모든 edge 후보를 그대로 쓰면 outlier 한두 개가 median을 끌고 간다. 두 가지 필터를 적용한다.

6.1 RTT 필터

edge 양 끝 샘플 중 한쪽이라도 RTT가 비정상적으로 크면 σ_edge가 폭주. 임계값:

RTT 임계값 = rttMedian × 1.5 + 2 (ms)

둘 다 통과해야 edge 후보 채택. (이 임계는 Select-LowJitterSamples와 동일.)

6.2 Gap 필터 (윈도우 간격)

RTT가 정상인데도 인접 샘플의 wall-clock 간격이 큰 경우가 있다 — 중간에 실패한 요청들이 silently 빠지면서 발생. 양 끝 RTT는 멀쩡한데 사이에 1~2초 timeout이 끼었던 케이스.

이런 edge는 균등분포 구간폭이 비정상적으로 커서 σ가 망가짐. 임계값:

간격 임계값 = (rttMedian + 50) × 3

이게 GUI의 분홍 갭 표시 기준과 동일. 분홍 띠 안의 edge는 자동으로 제외된다.

6.3 결과

필터 후 살아남은 edge들만 median 계산에 들어간다. 후보가 적어지는 대신 개별 σ가 작아진다.

7. 윈도우 연장

필터 후 edge 수가 너무 적으면(예: 4개 이하) 통계 신뢰도가 떨어진다. 메인 6초 윈도우가 끝나도 edge가 부족하면 추가로 샘플링한다.

  • Edge 8개 미만 → 추가 3초 윈도우
  • 여전히 8개 미만 → 또 3초 추가
  • 최대 3회 연장 → 측정 시간 최대 약 15초
  • 중간에 8개 채우면 즉시 종료

왜 무한 연장 안 하는가: hung이 잦은 사이트에서는 영원히 못 채울 수 있다. 3회 연장 후 그냥 가진 edge로 집계.

8. 시각 추정과 통계

8.1 각 edge가 offset 추정치 하나

필터 통과한 edge가 N개면 offset 추정치 N개:

candidates = [offset_1, offset_2, ..., offset_N]

8.2 최종 오프셋 채택 (기본: 1초 격자 교집합)

현재 기본은 각 edge가 주는 구간 제약 θ ∈ [S−R, S−L]의 교집합 중점이다(방법2). 과거의 "중점들의 median"(방법1)은 교집합이 비는 모순 상황의 폴백(edge-median)으로만 쓴다. 평균이 아니라 median/교집합을 쓰는 이유는 outlier 견고성이다. 두 방법의 상세 비교는 아래 13~14장을 본다.

기본: final_offset = midpoint( ∩ [S−R, S−L] )  |  폴백: median(midpoints)

8.3 신뢰구간 계산

95% 신뢰구간 반폭 (ci95) 계산:

σ = stddev(candidates)
ci95 = t × σ / √N

t값은 표본 수에 따라:

Nt값
≥ 301.96 (정규근사)
≥ 102.262
≥ 52.776
< 54.303

8.4 이론치 정밀도

σ_final ≈ 0.362 × √(1000(R+I)/N)

이 식은 두 가지 사실의 합성: (1) edge 하나의 σ ≈ (R+I)/√12, (2) median의 표준오차 ≈ 1.25σ/√N.

RTTCount이론 σ_final실측 추정 (1.5×)
50ms60±15ms±22ms
150ms30±25ms±37ms
300ms17±37ms±55ms

8.5 ci95의 한계

주의. ci95는 샘플 사이 흩어짐만 측정한다. 다음은 잡지 못한다:
  • RTT 비대칭으로 인한 체계적 bias
  • 서버측 Date 헤더 생성 타이밍 차이
  • PC NTP 자체 오차
  • CDN/프록시의 헤더 재작성
ci95 = ±49ms는 "측정이 일관됐다"는 시그널이지 "서버 시각과 49ms 이내"의 보장이 아니다.

또 한 가지: ci95 공식은 표본 평균용이다. median의 표준오차는 정규분포 가정 하 mean의 약 1.25배. 표시된 ci95는 median 기준으로는 ~25% 과소평가일 수 있다.

9. 재측정 정책

9.1 트리거

이 도구는 1분 자동 재측정을 하지 않는다.

  • 앱 시작 시 초기 측정 1회
  • 웹 UI에서 새 URL 입력 시 초기 측정
  • F5 새로고침 또는 페이지 재진입 시 재측정 1회

9.2 결과 검증 (기존 대상 재측정)

  1. accepted edge가 5개 미만 → 실패 처리, 기존 유지
  2. 재측정 결과가 기존 offset과 30ms 이하 → 미세 흔들림으로 보고 기존 유지
  3. 31~100ms 차이 → 새 측정값 반영
  4. 100ms 초과 → 한 번 더 재측정
  5. 2회차도 100ms 초과 → 새 값 버리고 기존 유지

9.3 새 URL 입력 시

대상이 바뀐 경우는 기존 offset과 비교 안 함. 서로 다른 서버 사이의 offset 차이는 정상 값.

10. 브라우저 보간

초기 측정은 최대 20초, 재측정은 최대 10초 안에서 수행되지만 화면은 매초 갱신해야 한다. 매번 측정 안 함. 기준점 하나를 잡고 브라우저 performance.now() 증가분으로 보간.

10.1 기준점 설정

측정 완료(status === 'ok') 시 /api/state 응답에서:

serverTimeAtSend = data.pcSendTimeAtMs + data.offsetMs
baseServerMs = serverTimeAtSend + (응답 받기까지 걸린 시간 / 2)
basePerfMs = performance.now()

10.2 매 프레임 갱신

현재 추정 = baseServerMs + (performance.now() - basePerfMs)

performance.now()는 monotonic이라 NTP 점프에도 영향 없다.

10.3 slew 보정

새 측정값이 도착해도 즉시 점프시키면 초침이 튄다. 작은 차이(120ms 이하)는 천천히 보정(slew). 큰 차이만 즉시 점프.

차이 > 120ms → 즉시 baseServerMs 교체
차이 ≤ 120ms → slewRemainingMs에 누적, 매 프레임 8% 비율로 반영

10.4 안전 마진 (적응형)

수강신청·티켓팅처럼 "절대 빠르면 안 되는" 용도를 위해 표시 시각을 안전마진만큼 의도적으로 늦춘다. 항상 실제보다 느리게 보여 클릭이 서버 기준보다 일찍 나가는 것을 막는다.

표시 시각 = 추정 서버 시각 − 안전마진,  안전마진 = max(30ms, RTT × 0.3)

왜 고정값이 아니라 RTT 비례인가:

  • 표시를 빠르게 미는 주범은 RTT 비대칭이다. 업로드(요청)가 다운로드(응답)보다 느리면 서버이벤트 추정이 실제보다 이르게 잡혀 offset이 커지고 표시가 빨라진다 — 가정용 인터넷에서 흔한 위험한 방향.
  • 이 비대칭 위험은 RTT에 대략 비례한다. 그래서 마진도 RTT의 30%로 비례시키고, 빠른 회선에서도 최소 30ms는 둔다. (RTT 90ms→30, 200ms→60, 400ms→120ms)
  • 마진은 14장의 핵심 한계와 짝을 이룬다: 교집합은 비대칭을 못 잡으므로, 잡지 못하는 그 편향을 마진으로 보수적으로 덮는다.
주의. 비대칭이 마진보다 크면(예: 업로드 심한 정체) 여전히 빠르게 표시될 수 있다. 마진을 키우면 안전하지만 평소엔 더 느리게 보이는 트레이드오프가 있다.

11. 네이버 특수 경로

11.1 왜 다른가

대부분 사이트는 HTTP Date 헤더가 유일한 시각 정보고 초 단위라 edge detection이 필요. 네이버는 검색 시계 위젯용 밀리초 정밀도 API가 별도로 있다.

https://ts-proxy.naver.com/dcontent/util/time.naver?passportKey=...&_format=yyyy/MM/dd/HH/mm/ss/SSS&site=naver

응답: nhntime("2026/05/23/23/26/59/195");

이 도구는 https://naver.com/이 입력되면 자동으로 이 경로를 쓴다.

11.2 passportKey 동적 갱신

API는 일회용 키를 요구한다. 키는 네이버 검색 결과 페이지 HTML에 박혀 내려오고 주기적으로 만료된다. 도구는 다음과 같이 처리:

  1. 첫 네이버 측정 시 search.naver.com/search.naver?query=시간을 fetch
  2. HTML에서 passportKey=([0-9a-f]{40}) 패턴 추출, 캐시
  3. 측정 중 API가 "Fail to get correct key" 반환 → 캐시 무효화 후 다음 시도에서 재발급

13. 방법1 vs 방법2

같은 edge 데이터에서 오프셋을 뽑는 두 방식. 방법1은 edge마다 구간 중점 (S−R + S−L)/2을 점추정하고 그 값들의 중앙값(median)을 쓴다. 방법2는 edge를 θ에 대한 구간 제약 θ ∈ [S−R, S−L]으로 보고 모든 구간의 교집합을 쓴다.

항목방법1 — 중점 median방법2 — 교집합(기본)
1초 격자 제약 사용안 씀(edge를 독립 측정 취급)씀(θ는 상수 1개)
오차 수렴∝ 1/√n∝ 1/n
불확실성통계적 신뢰구간(산포)feasible 영역 반폭(일치 폭)
outlier 견딤강함(median)순수 교집합은 1개에도 깨짐 → 하이브리드로 보완

13.1 왜 교집합이 더 좋은가

θ는 측정 내내 상수이고 서버 초 경계는 PC 시간축에서 정확히 1000ms 간격이다. 그래서 n개 edge는 "서로 다른 n개 값"이 아니라 하나의 θ가 동시에 만족할 n개 제약이다. 폭 ~140ms 구간이라도 교집합은 훨씬 좁아지고, 균등노이즈에서 이 교집합 중점이 MLE다(독일 탱크 문제). 오차가 median의 1/√n이 아니라 1/n로 준다.

13.2 워크드 예제 (진짜 θ=320ms, edge 3개)

edgeθ 구간 [S−R, S−L]중점
A[305, 445]375
B[185, 325]255
C[205, 345]275
 
A:                 [305===================445]
B:   [185=========325]
C:     [205==========345]
교집합:            [305-325]  → 중점 315
방법1 median(375,255,275) = 275 (오차 −45ms)  |  방법2 교집합 중점 = 315 (오차 −5ms, 폭 20ms)

edge 3개만으로도 오차가 1/9로 줄었다. 폭 140짜리 구간 3개의 교집합이 20으로 붕괴한 것이 핵심이다.

14. 하이브리드와 교집합의 한계

순수 교집합은 edge 하나가 어긋나면 통째로 밀리거나 공집합이 된다. 그래서 최대 겹침(interval stabbing)으로 가장 많은 edge가 공유하는 영역을 찾는 하이브리드를 쓴다.

상황method동작
모든 edge 일치edge-intersect전체 교집합 중점
일부 outlier 제외 후 일치(겹침 ≥2)edge-intersect-robust합의 부분집합 교집합, 나머지 버림
어떤 두 edge도 안 겹침edge-median방법1(중점 median)로 폴백
edge 0개upper-enveloperaw offset 상한봉투

14.1 ⚠️ 상호 일치 ≠ 정확

교집합은 랜덤한 불일치(지터·outlier)는 잡지만 공통 편향(RTT 비대칭)에는 눈이 먼다. 비대칭은 모든 edge 구간을 같은 방향으로 평행이동시켜 상호 일치를 유지한 채 집단을 편향시킨다.

몬테카를로(800회) 결과. 진짜 θ=320ms 기준:
시나리오edge-intersect 비율평균 오차coverage(보고 ± 안에 진짜 θ)
비대칭 0 (RTT 지터만)800/800~10ms100%
비대칭 +40ms800/800~40ms2%
비대칭 +80ms800/800~79ms0.4%
비대칭이 있어도 edge-intersect는 똑같이 100% 뜨면서 진짜 θ가 보고 ± 밖에 거의 항상 있다. 그래서 UI는 ✓ 같은 "성공" 표시를 쓰지 않고 "상호 일치(정확도 보장 아님)"로 적고, 안전마진을 max(30, RTT×0.3)로 키워 비대칭 위험에 비례시킨다.

즉 교집합은 방법1보다 정확하고 모순 진단까지 주지만, RTT 비대칭이라는 공통 편향은 두 방법 다 못 잡는 공통 한계다(외부 단방향 HTTP만으로는 관측 불가).
