1. 왜 이런 알고리즘이 필요한가?

특정 웹사이트(예: 인터파크, 네이버)의 정확한 서버 시각을 알아내고 싶다. 문제는 두 가지다.

  1. HTTP 응답의 Date 헤더는 초 단위만 알려준다. 밀리초는 직접 못 얻음.
  2. 요청을 보내고 응답을 받는 사이 네트워크 지연(RTT)이 있어서, 응답이 도착한 순간 헤더의 시각은 이미 과거다.
목표 정확도: ±50ms 이내. 네이비즘(±100~500ms)보다 더 정밀하게.

이를 달성하기 위해 여러 번 샘플링 → 통계 처리 → Edge(초 경계) 검출이라는 단계를 거친다.

2. 공통 빌딩 블록

초기 측정도 재측정도 핵심 엔진은 같다. 그 엔진의 이름은 Invoke-AdaptiveMultiSample이다.

2.1 1샘플이 하는 일

  1. t1: 요청 직전 PC 시각 기록 (모노토닉 클럭)
  2. 대상 서버에 HTTP HEAD 요청 (네이버는 시간 API)
  3. t2: 응답 직후 PC 시각 기록
  4. 응답의 Date 헤더 파싱 → Ts (서버가 응답을 보낸 시각, 초 단위)
  5. RTT = t2 - t1, raw offset = (Ts + RTT/2) − t2 계산
서버 추정 시각 ≈ Ts + RTT/2   (Cristian 알고리즘)

2.2 적응형 샘플링

1샘플로는 부정확하다. 여러 번 모은 다음 통계 처리해야 한다. 윈도우 길이는 약 6초로 고정한다.

Count = ceil(6000 / (RTT + 50))    [10 ≤ Count ≤ 60]
  • 샘플 간격: 50ms 고정 (서버 매너상 하한)
  • RTT가 작을수록 샘플을 많이 (최대 60개), 클수록 적게 (최소 10개)
  • 총 측정 시간은 어떤 환경에서든 약 6초로 일정

RTT가 매우 큰 환경(>500ms)에서는 MinCount=10 때문에 6초보다 길어질 수 있다. 의도된 동작.

2.3 Edge Detection (초 경계 검출)

Date 헤더는 초 단위라 ms를 직접 못 본다. 대신 헤더가 N초 → N+1초로 바뀌는 순간을 인접한 두 샘플 사이에서 포착한다. 이 경계 시각이 곧 서버의 정수 초 경계다.

검출된 edge들의 offset의 중앙값(median)을 최종 결과로 쓴다. (평균은 outlier에 약함)

왜 중앙값? 60개 샘플 중 몇 개가 패킷 지연으로 튀어도 중앙값은 거의 안 흔들린다.

2.4 정확도 이론치

σ_final ≈ 0.362 × √(1000 × (RTT + 50) / N)
RTT이론 σ_final실측 (≈1.5×)
50ms±15ms±20ms
150ms±25ms±35ms
300ms±37ms±55ms

RTT 200ms 이하에서 목표 ±50ms 달성.

3. 초기 측정 알고리즘최초 1회

프로그램이 처음 시작될 때, 또는 새 URL을 입력했을 때 딱 한 번 동작한다.

언제 실행되는가

  • ServerTimeProbe.exe 실행 시 URL이 주어진 경우 (즉시)
  • 브라우저에서 URL 입력창에 새 URL을 제출한 경우

흐름

  1. 준비: 상태 저장소 초기화, 대상 URL 설정
  2. RTT 추정 (3샘플): 첫 3개 샘플로 RTT median을 구한다
  3. 샘플 수 결정: Count = ceil(6000/(RTT+50)), [10, 60] 클램프
  4. 나머지 샘플 수집: 50ms 간격으로 Count개까지 채운다
  5. Edge 검출: 인접 샘플 사이 Date 초 전환을 찾아 edge offset 계산
  6. 중앙값 채택: edge offset들의 median이 최종 오프셋
  7. 저장: state.OffsetMs에 기록 → 시계가 동작 시작

4. F5 재측정 알고리즘F5 누를 때만

이미 초기 측정이 끝난 상태에서, 브라우저 새로고침(F5)이 일어났을 때만 동작한다. 1분마다 자동으로 돌지 않는다.

주의: 재측정은 시간이 오래 걸릴 수 있다 (최악의 경우 약 12초). 이는 의도된 안전 장치다 — 아래 5번 단계를 참고.

언제 실행되는가

  • 브라우저에서 F5 또는 페이지 재진입
  • 같은 URL을 다시 측정하는 경우

중요: 첫 페이지 로드는 재측정이 아니다. 두 번째 이후의 / 요청부터.

흐름

  1. 트리거: 서버가 / 요청을 받음 → MeasureTimer 시작 (100ms 후 발동)
  2. 1회차 측정: 빌딩 블록 그대로 적용 (약 6초)
  3. 편차 검사: 새 오프셋과 기존 오프셋의 차이를 계산
    |new − previous| ≤ 100ms 이면 → 수락, 종료
  4. 2회차 측정 (편차 큰 경우만): 한 번 더 6초 측정. 다시 차이 ≤ 100ms면 수락
  5. 거부 처리: 2회차도 100ms 초과면 → 새 값 버리고 기존 오프셋 유지, "재측정 편차 Nms 초과: 기존값 유지" 표시

왜 재시도/거부 장치가 있는가?

네트워크 일시적 흔들림이나 캐시/프록시 개입으로 한 번의 측정이 크게 튈 수 있다. 이때 새 값을 그대로 채택하면 시계가 갑자기 잘못된 시각으로 점프한다. 그래서:

  • 새 값이 기존과 크게 차이나면 의심한다.
  • 한 번 더 측정해서 같은 결론이 두 번 나오면 진짜 변화로 인정 (수락).
  • 그래도 안 맞으면 일시적 이상치로 판단하고 기존값을 지킨다.

클라이언트 측 처리 (clock.js)

  • 새 오프셋과 현재 표시값 차이 ≤ 120ms: 천천히 슬루잉(slewing)으로 부드럽게 보정
  • 차이 > 120ms: 즉시 점프
  • 목적: 측정값의 작은 흔들림이 초침의 위상 점프로 보이지 않게 함

6. 자주 묻는 질문

Q. 왜 자동으로 1분마다 재측정하지 않나?

과거 1분 자동 재측정 방식은 초침이 미세하게 흔들리는 위상지연을 유발했다. 사용자가 원할 때만 새로고침으로 재측정하도록 변경됨 (2026-05-02 수정).

Q. 6초 약속인데 실제로는 더 걸리는 것 같다.

두 가지 가능성:

  • 재시도 발동: 1회차 편차가 100ms를 넘으면 자동으로 2회차가 돌아 총 약 12초가 된다.
  • RTT가 매우 큼: 500ms 이상이면 MinCount=10 때문에 6초보다 길어진다.

Q. 클라이언트는 매초 서버에 요청을 보내나?

아니다. 1초마다 /api/state를 폴링해 상태만 확인한다. 시각 표시는 클라이언트의 performance.now()로 보간한다. F5를 누르지 않으면 측정 자체가 발생하지 않는다.

Q. NTP 정보는 어디에 쓰이나?

표시용. PC 시계가 NTP와 얼마나 어긋났는지 알려주는 참고치다. 오프셋 계산에는 사용하지 않는다.

Q. 네이버는 왜 다른가?

네이버는 자체 시간 API(ts-proxy.naver.com/dcontent/util/time.naver)가 ms 정밀도로 시각을 직접 알려준다. 그래서 Edge Detection이 필요 없고, 샘플들의 raw offset을 그대로 통계 처리한다. 적응형 샘플링 로직은 동일하게 적용.
