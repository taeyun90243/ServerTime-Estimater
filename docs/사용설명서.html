서버시간 측정 도구 - 사용설명서

Windows PowerShell 기반 로컬 GUI 도구. 대상 웹사이트의 서버 시각을 ±50ms 목표로 추정한다.

1. 설치와 실행

1.1 압축 풀기

release/ServerTimeTools.zip을 임의 폴더에 푼다. 풀린 폴더 안에는 다음이 있다.

  • ServerTimeProbe.exe - 실행 파일 (런처)
  • src/ - PowerShell 스크립트 모음 (수정하지 말 것)
  • docs/ - 이 문서를 포함한 설명 자료
  • addons/auto-clicker-gui/ - 별도 자동 클리커 도구

1.2 실행

ServerTimeProbe.exe를 더블클릭. 검은 PowerShell 콘솔창이 뜨고 기본 브라우저가 http://127.0.0.1:8765/를 자동으로 연다.

Tip. 콘솔창은 닫지 말 것. 닫으면 측정 서버가 종료된다. 측정이 끝나도 브라우저 시계는 계속 동작하지만, 재측정이나 새 URL 입력은 콘솔이 살아 있어야 가능하다.

1.3 종료

콘솔창에서 Ctrl+C 한 번. 또는 콘솔창 X 버튼. 자동으로 정리 후 종료된다.

2. 실행 요구사항

2.1 필수 환경

항목요구사항
운영체제Windows 10 또는 Windows 11
PowerShellWindows PowerShell 5.1 이상 또는 PowerShell 7.x
브라우저Microsoft Edge 또는 Chrome 계열 최신 브라우저 권장
네트워크HTTP/HTTPS 외부 접속 가능
  • Windows 10/11에는 Windows PowerShell 5.1이 기본 포함되어 있다.
  • run.bat은 PowerShell 7(pwsh)이 설치되어 있으면 우선 사용하고, 없으면 Windows PowerShell 5.1(powershell.exe)로 실행한다.
  • ServerTimeProbe.exe 런처는 Windows PowerShell(powershell.exe)로 src/probe.ps1을 실행한다.
  • 현재 측정 코드는 Windows PowerShell 5.1과 PowerShell 7.x 모두에서 동작하도록 작성되어 있다.

2.2 선택 환경

  • UDP 123 허용: NTP 참고 정보 표시용. 막혀 있어도 서버시간 측정 자체는 정상 동작한다.
  • PowerShell 7.x: 있으면 run.bat에서 우선 사용되지만 필수는 아니다.

2.3 필요 없는 것

  • Python 설치는 실행에 필요 없다.
  • ServerTimeClicker.exe도 빌드된 실행 파일을 쓰면 Python 설치가 필요 없다.
  • Python 3.10 이상과 PyInstaller는 EXE를 직접 다시 빌드할 때만 필요하다.

3. 처음 측정하기

  1. 브라우저 상단 입력창에 측정하고 싶은 URL을 넣는다. 예: https://ticket.interpark.com/
  2. 측정 버튼 클릭.
  3. 최대 20초 대기. 화면에 "초기 측정 중... (20초 이내)" 표시.
  4. 완료되면 시계가 동작 시작.

2.1 URL 형식

  • 스킴 생략 가능. example.com을 넣으면 https://example.com/로 자동 보정.
  • HTTPS 권장. HTTP만 지원하는 서버도 가능.
  • 사설 IP, localhost 같은 내부 주소는 거부될 수 있다 (SSRF 보호).

2.2 네이버 시계 비교

입력창에 https://naver.com/을 넣으면 일반 Date 헤더 대신 네이버 시계 API로 자동 전환된다. 네이버 검색의 시계 위젯과 같은 시각을 보여준다.

주의. "naver.com Date 헤더"와 "네이버 시계 API"는 다른 값이다. 자세한 차이는 알고리즘_설명.html의 "네이버 시계와 Date 헤더의 차이" 절 참고.

4. 화면 읽는 법

3.1 큰 시계

중앙에 HH:MM:SS 형식. 초가 녹색. 바깥 원형 progress와 시침이 1분 단위로 한 바퀴 돈다. 12개 tick은 5초마다 하나씩 켜진다.

3.2 상태 표시 (시계 아래)

상태의미
측정할 URL을 입력하세요idle. URL 미입력.
측정 대기 중...queued. 곧 측정 시작.
초기 측정 중... (20초 이내)측정 버튼으로 새 측정 진행.
재측정 중... (15초 이내)F5 새로고침으로 재측정 진행.
재측정 차이 Nms: 기존값 유지기존 측정과 30ms 이하 차이라 미세 흔들림을 반영하지 않음.
재측정 완료 (2회차 반영)1회차 편차 컸지만 2회차에서 안정.
재측정 편차 Nms 초과: 기존값 유지2회 시도 모두 100ms 넘게 튀어 기존값 유지.
측정 실패대상 서버가 Date 헤더를 안 주거나 5초 timeout.

3.3 통계 줄

측정: 12초 전  RTT 63ms  ±49ms  샘플 38개 (edge 4개)  안전마진 -30ms
  • 측정: N초 전 - 마지막 측정 완료 후 경과 시간
  • RTT 63ms - 측정 중 왕복 지연 중앙값
  • ±49ms - 95% 신뢰구간 반폭. 작을수록 측정이 안정적.
  • 샘플 N개 (edge M개) - 보낸 총 샘플 수 / 그중 사용된 edge 수
  • 안전마진 -30ms - 표시 시각을 의도적으로 30ms 과거로 표시 (네트워크 지연 보정용 마진)

3.4 NTP 정보 줄

참고: PC 시계 +12ms - PC 시계가 NTP(KRISS) 대비 +12ms 빠르다는 의미. 측정값에는 영향 없고, PC 자체 시계 상태 진단용.

읽기 팁. ±값이 50ms 이하면 측정 안정. 100ms 이상이면 노이즈 큼 → F5로 재측정 권장. ±300ms 이상이면 결과 신뢰 불가.

5. F5 재측정

시간이 지나면 PC 시계와 서버 시계 사이 작은 드리프트가 누적될 수 있다. 정확도를 다시 끌어올리려면 브라우저에서 F5 새로고침.

4.1 동작

  1. F5 누르면 백그라운드에서 동일한 6~15초 적응형 재측정.
  2. 새 offset이 기존과 100ms 이내 차이면 즉시 반영.
  3. 100ms 초과면 한 번 더 측정. 2회차도 초과면 새 값 버리고 기존 유지.
주의. 재측정은 6초 이상 걸린다. 티케팅 직전 1분 이내에는 F5 누르지 말 것. 측정 중에는 표시 시각이 변할 수 있다.

4.2 자동 재측정 없음

이 도구는 1분 자동 재측정을 하지 않는다. F5만이 재측정 트리거. 짧은 세션(수분 이내) 동안에는 처음 측정값을 그대로 쓰면 됨.

6. 측정 상세 보기 (시각화)

화면 하단의 측정 상세 보기 버튼을 누르면 마지막 측정의 샘플 분포를 타임라인으로 본다.

5.1 점의 의미

색의미
녹색샘플의 Date 헤더 초 값이 짝수
파란색샘플의 Date 헤더 초 값이 홀수
오렌지RTT가 중앙값의 2배 이상 (느린 샘플)

녹색→파란색 또는 파란색→녹색으로 색이 바뀌는 인접 점 사이에 빨간 점선 (edge)이 그려진다. 그게 Date 초가 N→N+1로 넘어간 경계.

5.2 분홍 띠 (gap-band)

인접 샘플의 wall-clock 간격이 정상보다 3배 이상 벌어진 구간. 보통 timeout이나 일시적 연결 실패. 이 띠 안의 빨간 edge는 알고리즘에서 자동으로 제외된다.

5.3 x축

측정 시작 후 경과 시간(PC 시계 기준). 점은 응답이 도착한 시각(t2)에 그려진다. 송신 시각(t1)이 아니다.

7. 자동 클리커 (선택)

addons/auto-clicker-gui/dist/ServerTimeClicker.exe는 서버시간 도구가 측정한 시각에 맞춰 현재 마우스 위치를 자동 클릭하는 도구다. 별도 실행.

6.1 기본 사용

  1. 먼저 ServerTimeProbe.exe를 실행해 측정 중인 상태로 둔다.
  2. ServerTimeClicker.exe 실행.
  3. 목표 시각(예: 20:00:00.000) 입력.
  4. 마우스를 클릭할 위치에 두고 시작 버튼.
  5. 지정 시각에 자동 클릭.

6.2 보정 파라미터 (기본값 권장)

  • Lead ms = 0 - 목표보다 몇 ms 앞당겨 클릭할지. 실제 환경에서 늦으면 양수로 보정.
  • Resync before ms = 3000 - 목표 N ms 전에 서버 시간 도구 상태를 다시 읽어 기준점 재설정.
  • Final spin ms = 25 - 마지막 25ms는 sleep 없이 busy-wait로 정밀도 확보.
주의. 클릭 정확도는 측정 자체뿐 아니라 Windows 스케줄러 지연, 브라우저 입력 처리 지연, 서버 전송 지연이 누적된다. 실측은 사용 환경에서 직접 보정해야 한다.

8. 문제 해결

7.1 "측정 실패" 메시지

  • 대상 서버가 Date 헤더를 주지 않는 경우. 다른 URL 시도.
  • 네트워크 끊김 또는 DNS 실패.
  • 대상 서버가 HEAD/GET 모두 5초 안에 응답 없음.

7.2 측정이 30초 이상 걸림

  • 대상 사이트가 PHP/zboard 같은 구형 서버라 HEAD 요청에서 hang 발생. 자동 timeout으로 진행은 되지만 시간이 늘어남.
  • RTT가 매우 큰 해외 사이트.
  • 측정 중 timeout이 잦으면 ±값이 커진다. 안정적 사이트로 바꾸거나 그대로 사용.

7.3 F5 눌렀는데 "초기 측정 중"으로 표시

브라우저 캐시 문제. Ctrl+F5로 강제 새로고침. 그래도 안 되면 PowerShell 콘솔창 닫고 재실행.

7.4 시계가 이상하게 빨라지거나 느려짐

매우 드물지만 PC NTP 동기화가 점프를 일으키면 발생 가능. 도구는 monotonic clock(performance.now())을 쓰지만 초기 기준 anchor에는 PC 시계를 쓴다. F5 재측정으로 회복.

7.5 ±값이 200ms 이상 나옴

  • RTT가 매우 큰 사이트 (해외, 300ms+).
  • 대상 서버가 Date 헤더를 캐싱하거나 CDN에서 재작성하는 경우.
  • 측정 중 timeout이 여러 번 발생한 경우.
  • F5 한 번 더 시도. 그래도 안 줄어들면 그 사이트는 본질적으로 그 정도 정확도가 한계.

9. 한계와 주의사항

8.1 정확도의 의미

표시되는 ±값(ci95)은 측정 내부의 일관성이지 실제 정확도가 아니다. 다음 요인은 ci95에 안 잡힌다.

  • RTT 비대칭 (송신과 수신 지연이 다른 경우)
  • 서버측 Date 헤더 생성 시점 (요청 받자마자 vs 응답 직전)
  • PC NTP 자체 오차
  • CDN/프록시가 Date 헤더를 재작성하는 경우

8.2 정확도가 좋은 조건

  • 국내망 + RTT 안정 (50-150ms)
  • 대상이 직접 응답하는 서버 (CDN 캐시 없음)
  • 네이버 시계 API 같은 ms 정밀도 API 사용

8.3 티케팅에 쓸 때

티케팅의 오픈 판정은 "실제 예매 요청을 받는 서버"의 시각이다. 그 서버 URL을 직접 측정하는 게 가장 정확하다. naver.com이나 네이버 시계 API는 참고 기준일 뿐 실제 예매 서버와 같다는 보장이 없다.

8.4 보안

  • 로컬 도구다. 다른 사용자의 PC에 노출되지 않는다.
  • 측정 요청은 사용자가 입력한 URL로만 나간다. 제3자에게 데이터 전송 없음.
  • 로그는 logs/probe-YYYYMMDD.jsonl에 JSONL 형식으로 쌓인다. 30일 이상 된 로그는 자동 삭제.
